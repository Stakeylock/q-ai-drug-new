# Phase 16A — Report APIs Documentation

## Overview

Phase 16A introduces the **backend report data model and API layer** for QuDrugForge.

> **Nothing is generated in this phase.**
> All routes operate on metadata only. Actual PDF/CSV/SDF/HTML generation is Phase 16B.

## Report Types

| `report_type`       | Purpose |
|---------------------|---------|
| `project_summary`   | Full project report across all available data |
| `candidate_dossier` | Detailed dossier for selected candidate molecules |
| `experiment_report` | Report scoped to a single docking/GNINA/quantum/ADMET experiment |
| `imported_q_ai_drug`| Imported q-ai-drug report.pdf/report.html artifacts |
| `custom`            | Future manual/custom report |

---

## Report Statuses

| Status       | Meaning |
|--------------|---------|
| `draft`      | Metadata created, no generated file |
| `queued`     | Generation requested, not yet started |
| `generating` | Report generation in progress (Phase 16B) |
| `completed`  | Generated by QuDrugForge (Phase 16B) |
| `failed`     | Generation failed |
| `imported`   | Existing artifact imported (q-ai-drug or manual) |

## Implemented Routes

All routes require a valid JWT Bearer token. All enforce project/workspace membership.

| Method   | Path                                                            | Description |
|----------|-----------------------------------------------------------------|-------------|
| `GET`    | `/api/v1/projects/{project_id}/reports/summary`                 | Dashboard summary + section availability |
| `POST`   | `/api/v1/projects/{project_id}/reports/import-q-ai-drug`        | Register q-ai-drug artifact as report |
| `GET`    | `/api/v1/projects/{project_id}/reports`                         | List reports (filterable) |
| `POST`   | `/api/v1/projects/{project_id}/reports`                         | Create draft report |
| `GET`    | `/api/v1/projects/{project_id}/reports/{report_id}`             | Fetch report metadata |
| `PATCH`  | `/api/v1/projects/{project_id}/reports/{report_id}`             | Update safe fields |
| `DELETE` | `/api/v1/projects/{project_id}/reports/{report_id}`             | Delete report |
| `POST`   | `/api/v1/projects/{project_id}/reports/{report_id}/queue-generation` | Queue for Phase 16B generation |
| `GET`    | `/api/v1/projects/{project_id}/reports/{report_id}/files`       | Get file metadata + download URLs |

> **Route ordering:** `/summary` and `/import-q-ai-drug` are declared _before_ `/{report_id}` in the router to prevent FastAPI from misrouting them as report_id values.

## Section Availability Logic

The report service inspects real MongoDB collection counts:

| Section       | Available if... |
|---------------|-----------------|
| `overview`    | Always (project exists) |
| `targets`     | `targets` collection has documents for project |
| `candidates`  | `molecules` collection has documents |
| `docking`     | `docking_results` collection has documents |
| `gnina`       | `gnina_results` collection has documents |
| `quantum`     | `quantum_results` collection has documents |
| `admet`       | `admet_results` collection has documents |
| `simulations` | `simulation_results` collection has documents |
| `artifacts`   | `files` collection has documents |

Sections requested but missing data → `"status": "missing"` with explanation summary.

---

## How Imported q-ai-drug Reports Are Represented

When `POST /reports/import-q-ai-drug` is called:

1. If `file_ids` are provided explicitly → validate they belong to the project.
2. If `file_ids` is empty → auto-discover by scanning `files` collection for:
   - `source_module` in `["q_ai_drug", "reports", "q_ai_drug_import"]`
   - `file_type` in `["generated_report", "imported_report", "report", "q_ai_drug_artifact"]`
   - `original_filename` contains `"report"` and ends with `.pdf` or `.html`
3. Deduplication: if same `source_output_dir` + project already has a record → returns existing.
4. `primary_file_id` prefers PDF over HTML.
5. Report document gets:
   - `report_type: "imported_q_ai_drug"`
   - `status: "imported"`
   - `source: "q_ai_drug"`

## Example Requests & Responses

### Create draft project_summary report

```bash
curl -X POST http://localhost:8001/api/v1/projects/PROJECT_ID/reports \
  -H "Authorization: Bearer TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "EGFR NSCLC Full Discovery Report",
    "report_type": "project_summary",
    "sections_requested": ["overview","targets","candidates","docking","gnina","quantum","admet","simulations"]
  }'
```

**Response:**
```json
{
  "success": true,
  "data": {
    "report_id": "a1b2c3d4-...",
    "project_id": "...",
    "title": "EGFR NSCLC Full Discovery Report",
    "report_type": "project_summary",
    "status": "draft",
    "source": "qudrugforge",
    "sections": [
      {"section_id": "overview", "status": "available", "summary": ""},
      {"section_id": "targets",  "status": "available", "summary": ""},
      {"section_id": "docking",  "status": "missing",   "summary": "No backend data available for this section yet."}
    ],
    "metadata": {"candidate_count": 214, "has_docking": false, ...},
    "file_ids": [],
    "primary_file_id": null,
    "created_at": "2026-05-18T12:30:00"
  },
  "message": "Report draft created successfully"
}
```
