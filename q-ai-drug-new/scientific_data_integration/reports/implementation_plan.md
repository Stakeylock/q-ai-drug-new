# Implementation Plan — Scientific Hardening

This plan outlines the systematic implementation of the hard scientific features specified in `docs/LLM_SCIENTIFIC_HARDENING_PLAYBOOK.md` to harden the scientific validity of the platform's computational hypotheses.

## User Review Required

All outputs generated by these runners represent **computational hypotheses only** and must not make any definitive therapeutic claims. The terminology guidelines in Part I of the playbook will be strictly enforced across all output artifacts.

> [!IMPORTANT]
> - All fallback/mock/missing evidence must be explicitly labeled as such in result columns (e.g. `docking_is_real=False`, `is_heuristic_fallback=True`).
> - We will avoid overclaiming or labeling simulated/proxy data as biochemically proven.

## Proposed Changes

### Component 1: Applicability Domain Guard
Improve `ApplicabilityDomainGuardRunner` in [downstream.py](file:///E:/q-ai-drug-new/src/q_ai_drug/product/module_runners/downstream.py) to calculate Tanimoto nearest-neighbor similarities using RDKit's Morgan 2048-bit fingerprints when a training set is provided, falling back to QED/alert proxy scores when RDKit/training sets are missing.

### Component 2: Activity Model Studio
Enhance `ActivityModelStudioRunner` in [downstream.py](file:///E:/q-ai-drug-new/src/q_ai_drug/product/module_runners/downstream.py) to:
- Use ECFP Morgan fingerprints when available.
- Train both `RandomForestRegressor` and `ExtraTreesRegressor`.
- Perform scaffold-based split using RDKit when a custom split column is absent.
- Record comprehensive performance metrics including $R^2$, RMSE, MAE, train/test sizes, and fingerprint specs.

### Component 3: Redocking Validation
Connect the RDKit-based RMSD computation helper in [redocking.py](file:///E:/q-ai-drug-new/src/q_ai_drug/docking/redocking.py) to the Q-Dock execution path in [q_dock_studio.py](file:///E:/q-ai-drug-new/src/q_ai_drug/product/module_runners/q_dock_studio.py), writing exact RMSD values and validation outcomes to `redocking_validation.csv`.

### Component 4: Interaction Fingerprints
Integrate the geometric contact classification from [interactions.py](file:///E:/q-ai-drug-new/src/q_ai_drug/docking/interactions.py) into Q-Dock execution in [q_dock_studio.py](file:///E:/q-ai-drug-new/src/q_ai_drug/product/module_runners/q_dock_studio.py), capturing residue-level polar, hydrophobic, and salt-bridge contacts when PDB receptors and SDF poses are available.

### Component 5: Standalone GNINA execution
Wire the standalone GNINA subprocess runner from [gnina_runner.py](file:///E:/q-ai-drug-new/src/q_ai_drug/docking/gnina_runner.py) into Q-Dock, executing CPU-based GNINA docking when requested and when the binary is available, producing parseable CNN scores and CNN affinity values.

### Component 6: Integration and Verification
Add a comprehensive Golden Scientific Chain test in `tests/test_scientific_golden_chain.py` verifying end-to-end telemetry from raw CSV candidates up to the evidence fusion and final claim matrix report.
